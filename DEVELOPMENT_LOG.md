## Phase 1: Data Architecture & Strict Validation

### Target

To ensure that all data sources—including system logs, manually labeled ground truths, and simulation data—adhere to a consistent, and pre-defined structure

### Data pipeline

- **Input**: Read raw, unstructured, or semi-structured data logs from `sample_rag_logs.jsonl`
- **Process**: 
    - **Parsing**: Extract core fields required for evaluation (`query`, `context`, `response`)
    - **Mapping**: Mapping raw data to a standardized schema structure
    - **Validation**: Validate and clean input data
    - **Mode Detection**: Identify whether the record belongs to an *Offline* or *Online* evaluation flow based on the presence of `expected_context_ids`
- **Output**: Export the validated, strongly-typed dataset to `evaluation_records.parquet`

### Tasklist

1. Create a schemas.py using Pydantic to define the core data contract
2. Create sample_rag_logs.jsonl (contains diverse edge cases)
3. Write `data_loader.py` to read JSONL file
4. Write validator.py to validate each record
5. Write normalizer.py to normalize field formats
6. Write mode_detector.py to define evaluation mode
7. Export the sanitized dataset to evaluation_records.parquet

### Output

| Ensure a consistent "Data Contract" across the entire Pipeline

## Phase 2: Local Metrics

### Target

Calculate system performance metrics `System Metrics` and retrieval layer quality metrics `Retrieval Metrics` at the local level, independent of LLM Evals.

### Data Pipeline

- **Input**: Read the normalized data from `evaluation_records.parquet`
- Process and calculate the metrics, including: `latency`, `token usage`, `cost estimate`, `Hit Rate@K`, `MRR`, `Recall@K`, `Precision@K`
- **Output**: Export the data component file `local_metric_results.parquet`, and the summary report in phase 2 at file `local_metric_summary.json`

### Tasklist

1. Add metric schemas using Pydantic.
2. Create a configuration file `metrics_config.yaml` (defines token pricing per model)
3. Write a loader method to read data from `evaluation_records.parquet`
4. Write a module `retrieval_metrics.py` (computes the system metrics value)
5. Write modules `cost_metrics.py` / `system_metrics.py` 
6. Write `local_metric_calculator.py` (orchestrate the pipeline and compute metrics for all records)
7. Export `local_metric_results.parquet` and `local_metric_summary.json`
8. Write unit tests for each metric to handle edge cases

### Architectural Notes & Constraints

> ⚠️ **Trigger Condition for Retrieval Metrics (`Hit Rate@K`, `MRR`, `Recall@K`, `Precision@K`):**
> These metrics are **only activated** for calculation if the record contains the `expected_context_ids` field (Ground Truth). If this field is empty or `null`, the system will gracefully skip these metrics (returning `nan`) instead of throwing a runtime exception.

#### What is `expected_context_ids` and where does it come from?
This is a list of IDs for the standard document segments corresponding to user questions. The presence of this field depends entirely on the evaluation mode:

* **Offline Evaluation (Simulation / Test Environments):** **ACTIVATED**. The data comes from a curated **Golden Dataset** (either manually labeled by human experts or generated by an LLM based on existing knowledge bases) to benchmark retrieval algorithms before deploymen.

* **Online Evaluation (Production Logs):** **DEACTIVATED**. Real-world chatbot logs generated by end-users will **not** have this field, as user questions are unpredictable and lack predefined answers. In this scenario, the pipeline skips local retrieval metrics and defers to **LLM-as-a-judge** in Phase 3.

*Note on Parameter `K`:* `K` is the maximum number of contexts that the Retrieval layer can retrieve to throw into the prompt for the LLM to generate a response

### Outputs

**`local_metric_results.parquet`**: Detailed, per-record metrics useful for deep-dive debugging and error analysis
**`local_metric_summary.json`**: An aggregated report containing averaged metrics (Average Latency, Total Cost, Mean MRR, etc.) used for experiment tracking and system benchmarking

## Phase 3: Judge-Based Metrics

### Target

Using Ragas/DeepEval or LLM judge to in-depth evaluate quality of answer and retrieved context

### Data pipeline

- **Input**:
    - Read the normalized data from `evaluation_records.parquet`
    - Using the result report at phase 2 from `local_metric_results.parquet`
- **Process**: Calculate the metrics, including:
    - **Generation quality**: `faithfulness`, `answer_relevancy / response_relevancy`, `answer_correctness`(if `ground_truth` exists), `hallucination_risk / hallucination_flag`
    - **Retrieval judge-based quality**: `context_relevance`, `llm_judged_hit_rate@K` (if expected_context_ids not exist), `llm_judged_context_precision@K`
- **Output**: Export the data component file `judge_metric_results.parquet`, and the summary report in phase 2 at file `judge_metric_summary.json`, `failed_judge_samples.jsonl`

### Tasklist

1. Add judge metric schemas (use required fields for Ragas/DeepEval)
2. Create judge_config.yaml
3. Create judge cache layer
4. Write adapter to nomalize Ragas/DeepEval' input
5. Implement Ragas evaluator
6. Implement DeepEval evaluator
7. Implement custom LLM judge fallback
8. Compute judge metrics each record
9. Export judge_metric_results.parquet + summary JSON
10. Writh unit tests + smoke test

> Note: At phase 3, focus on using and evaluating with `Ragas`, `DeepEval`, `LLM-as-a-judge`. There are third-party fees incurred (OpenAI, Gemini,..)
